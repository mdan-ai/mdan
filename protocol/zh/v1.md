# MDAN 协议 v1

MDAN 协议 v1 定义了 `text/markdown` 文档如何承载 MDAN 的交互页面语义。

当客户端收到这个 profile 时，它不应该把响应当成一篇只读 Markdown 文章，而应理解为：这是一个 canonical 的 MDAN 页面或片段响应，其中可能包含可执行交互结构。

## Media Type

MDAN 协议 v1 的声明形式是：

```http
Content-Type: text/markdown; profile="https://mdan.ai/protocol/v1"
```

基础 media type 仍然是 `text/markdown`。

`profile` 参数的作用，是告诉客户端这份 Markdown 承载的是 MDAN 协议语义。

## 这意味着什么

带有这个 profile 的响应可能包含：

- 一份 canonical 页面或片段 Markdown
- 一个可执行的 `mdan` fenced block
- `BLOCK` 声明
- `INPUT` 声明
- `GET` 和 `POST` 操作声明
- 通过 `<!-- mdan:block ... -->` 建立的 block 锚点映射
- 下一步交互所需的上下文

也就是说，这不只是“读完就结束”的内容，而是可能继续驱动交互的协议文本。

## 客户端最少应该怎么处理

收到这个 profile 的客户端至少应该：

- 把这份 Markdown 当成 canonical 的 MDAN 内容
- 保留可执行 `mdan` 代码块和 block 锚点
- 理解返回值里可能不只是当前状态，也可能包含下一步交互
- 当 HTML 和这份 Markdown 同时存在时，优先把这份 Markdown 视为协议真源

## 覆盖范围

协议 v1 当前覆盖：

- 完整页面 Markdown 响应
- 片段 Markdown 响应
- 基于 HTTP `GET` / `POST` 的交互
- 同一个应用的 Markdown / HTML 双表示

协议 v1 当前不定义：

- HTTP 之外的传输方式
- 浏览器渲染细节
- 身份认证策略
- 非 HTTP 运行环境

## 与 HTML 的关系

HTML 是同一个应用在浏览器侧的承载投影。

当 Host 返回 HTML 时，应该通过下面这些信号帮助 Agent 发现 canonical Markdown：

- `rel="alternate" type="text/markdown"`
- 如果站点明确提供了真实的 `llms.txt`，可以额外输出 `rel="llms-txt"`
- 可选的等价 `Link` 响应头

真正的 profile 声明属于 Markdown 响应本身，而不是 HTML 响应。

## 版本约定

`v1` 是 MDAN 的第一版公开协议 profile。

未来如果协议语义发生版本升级，应使用新的稳定 profile URL，而不是偷偷改变 `v1` 的含义。
