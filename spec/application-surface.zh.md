# MDAN 应用表面规范

- 状态：Draft
- 版本：vNext
- 语言：中文

## 1. 引言

本规范定义 MDAN 应用表面的规范性语义。

目标是定义一套与具体实现解耦的应用表面模型，使不同实现能够围绕相同的内容语义、动作语义、状态语义、表示语义与安全语义建立互操作性。

本规范定义：

- 内容文档的规范对象与结构约束
- 动作契约的规范对象与结构约束
- 状态绑定表面的语义
- 动作执行、状态迁移与结果分类
- 表示、投影与协商的一致性要求
- 信任、安全、确认与合规要求

本规范不定义：

- 任意特定 SDK 的 API
- 任意特定编程语言或框架
- 任意特定宿主或运行时的内部实现方式
- 任意特定 UI 组件库或交互外观
- 教程式使用方法、接入步骤或最佳实践

## 2. 规范性说明

### 2.1 规范性关键字

本规范中的下列关键字按其规范性含义解释：

- `MUST` / `必须`：满足合规所必需
- `MUST NOT` / `不得`：满足合规所必需的禁止项
- `SHOULD` / `应当`：强烈建议；若偏离，必须有明确理由
- `SHOULD NOT` / `不应`：强烈不建议；若偏离，必须有明确理由
- `MAY` / `可以`：可选能力

### 2.2 解释优先级

若两个规范语句发生冲突，解释优先级如下：

1. 更具体的规则优先于更一般的规则
2. 更严格的安全约束优先于更宽松的安全约束
3. 动作契约中的执行真相优先于内容文档中的叙述性表达
4. 本规范优先于任何实现文档、SDK 文档或示例文档

## 3. 核心术语

### 3.1 内容文档

内容文档是面向人类与代理共同阅读的规范文本载体。它定义可读内容、结构化区域以及动作引用语义，但不单独构成完整的执行真相。

### 3.2 动作契约

动作契约是机器可判定的执行契约。它定义状态身份、动作集合、输入模式、状态影响、允许的后继动作以及相关安全语义。

### 3.3 表面

表面是绑定到某一具体状态的运行时表面。它至少由内容文档、动作契约与状态身份共同确定，并可以被投影为不同表示形式。

### 3.4 状态

状态是应用某一时刻的身份边界。状态至少由 `app_id`、`state_id` 和 `state_version` 共同标识。

### 3.5 区域

区域是表面中可被单独识别和更新的命名部分。区域更新与整页更新在语义上必须区分。

### 3.6 动作

动作是可执行的状态相关操作。每个动作必须具有稳定身份，并具有明确的目标、输入约束、状态影响和安全语义。

### 3.7 表示

表示是同一状态表面的可交付形式。不同表示可以面向不同消费方，但不得改变同一状态的规范真相。

### 3.8 信任

信任是对某一内容区域解释权限和安全语义的声明。更保守的信任解释优先。

### 3.9 合规

合规是指某一实现满足本规范中对对象结构、状态一致性、执行语义、安全语义和结果分类的最低要求。

## 4. 总体模型

一个合规的 MDAN 应用表面模型由以下核心对象构成：

- 内容文档
- 动作契约
- 状态绑定表面

这些对象之间满足以下不变量：

- 内容文档必须承担共享阅读表面的职责
- 动作契约必须承担执行真相的职责
- 表面必须绑定到一个明确状态
- 状态不匹配不得被静默合并
- 表示差异不得改变同一状态的规范语义

内容文档、动作契约和表面之间是关联关系，而不是互相替代关系。

## 5. 内容文档模型

### 5.1 一般要求

内容文档必须是文本形式的规范内容载体。

内容文档可以包含：

- 普通 Markdown 内容
- 结构化 block 区域

内容文档不得单独承担以下职责：

- 定义完整的动作执行真相
- 取代状态身份字段
- 覆盖动作输入模式

### 5.2 Block 区域

内容文档可以声明命名 block 区域。

block 区域的规范形式为：

```md
::: block{id="main" actions="refresh_main,submit_message" trust="untrusted"}
...
:::
```

block 区域满足以下要求：

- `id` 必须存在，且必须是非空字符串
- 同一内容文档内 block `id` 必须唯一
- `actions` 可以省略；若存在，必须是由逗号分隔的动作标识列表
- `trust` 可以省略；若存在，其值必须是 `trusted`、`untrusted` 或 `unknown` 之一
- block 正文不得为空

block 区域中的每个动作引用都必须满足：

- 引用的动作标识必须存在于同状态的动作集合中
- 同一 block 的 `actions` 列表中不得重复引用同一动作标识

### 5.3 语义槽位

语义槽位是内容文档中的结构能力。

本规范承认的槽位名称为：

- `Purpose`
- `Context`
- `Rules`
- `Result`
- `Views`
- `Handoff`

若某一实现声明支持语义槽位，则：

- 槽位标题必须可判定
- 重复槽位名称不得被接受
- 空槽位正文不得被接受

### 5.4 代理区块

代理区块用于承载 agent-only 内容。

规范形式为：

```md
<!-- agent:begin id="login_hint" -->
...agent-only Markdown...
<!-- agent:end -->
```

若实现承认代理区块语义，则：

- 每个 `agent:begin` 必须与一个 `agent:end` 配对
- 每个代理区块必须具有非空且唯一的 `id`
- 空代理区块正文不得被接受
- 嵌套代理区块不得被接受

### 5.5 内容文档一致性

内容文档中的动作引用满足以下一致性要求：

- block 区域中的结构化动作引用必须指向已声明动作

若内容文档与动作契约不一致，则内容文档不得被解释为覆盖动作契约。

## 6. 动作契约模型

### 6.1 总体结构

动作契约必须是可判定的对象结构。

动作契约对象的顶层至少必须包含：

- `actions`

动作契约对象可以包含：

- `app_id`
- `state_id`
- `state_version`
- `blocks`
- `actions`
- `allowed_next_actions`
- `security`

### 6.2 状态身份字段

若动作契约声明了状态身份字段，则必须满足：

- `app_id` 必须是非空字符串
- `state_id` 必须是非空字符串
- `state_version` 必须是数值

同一状态表面中的内容、动作和派生表面不得在状态身份字段上互相矛盾。

### 6.3 区域声明

`blocks` 可以用于声明该状态中可识别的区域集合。

若存在 `blocks`，则：

- 每个区域标识必须是非空字符串
- 同一集合中不得重复

### 6.4 动作集合

`actions` 必须是动作对象数组。

每个动作至少必须具有：

- `id`
- `target`

动作 `id` 在同一状态中必须唯一。

### 6.5 允许的后继动作

`allowed_next_actions` 若存在，则：

- 必须是动作标识数组
- 数组中的每一项都必须在 `actions` 中存在
- 消费者不得执行未出现在该列表中的动作

## 7. 表面模型

### 7.1 表面定义

表面是内容文档、动作契约和状态身份共同确定的状态绑定结果。

一个合规表面至少必须满足：

- 具有明确状态身份
- 具有可读内容语义
- 具有可执行动作语义
- 具有与状态影响一致的更新边界

### 7.2 表面种类

本规范承认以下表面结果种类：

- `page`
- `region`
- `stream`

它们必须在语义上相互区分：

- `page` 表示整页状态结果
- `region` 表示区域级状态结果
- `stream` 表示持续交付的增量结果流

实现不得将三者静默折叠为同一结果类别。

### 7.3 表面附加元数据

表面可以携带附加元数据，只要这些字段：

- 不覆盖状态身份
- 不改变动作执行真相
- 不改变结果类别

若表面声明区域结果映射，则：

- 每个键必须是非空区域名称
- 每个值必须是该区域在当前状态中的内容字符串

## 8. 动作执行与状态迁移

### 8.1 动作执行

动作执行必须基于当前状态中的已声明动作。

消费者：

- 必须使用已声明动作的 `id`
- 必须遵守目标与方法语义
- 必须遵守输入约束
- 必须遵守允许的后继动作限制

### 8.2 执行结果

成功的动作执行可以产生：

- 新的整页表面
- 区域级更新表面
- 流式结果

结果必须保持状态、动作与可读内容之间的一致性。

### 8.3 失败结果

若动作无法执行：

- 返回结果应尽可能保持可读
- 返回结果不得暗示一个被拒绝的动作已经成功
- 实现可以收窄或清空 `allowed_next_actions`

## 9. 表示与交付

### 9.1 表示种类

本规范承认以下表示类别：

- Markdown surface representation
- HTML
- Event Stream

其它表示可以服务于不同消费方，但它们必须共同指向同一状态语义。

### 9.2 表示一致性

表示之间必须满足以下要求：

- 不得因表示不同而改变状态身份
- 不得因表示不同而改变动作集合
- 不得因表示不同而改变安全约束
- 不得因表示不同而改变区域更新语义

Markdown surface representation 承担共享阅读表面；其它表示是同一状态的投影，不得改变其规范语义。

### 9.3 协商

若存在表示协商，则实现必须根据显式协商结果返回对应表示，或显式拒绝不可接受的表示请求。

实现不得在协商失败时静默返回与请求语义冲突的表示。

## 10. 信任与安全模型

### 10.1 信任取值

本规范承认以下信任取值：

- `trusted`
- `untrusted`
- `unknown`

### 10.2 安全边界

更保守的信任解释优先于更宽松的信任解释。

未明确声明为可信的内容，不得被解释为可以放宽结构、执行或安全约束。

## 11. 合规要求

一个合规实现至少必须满足：

- 能够表达内容文档
- 能够表达动作契约
- 能够表达状态身份
- 能够区分 page、region、stream 三类结果
- 能够在不同表示之间保持语义一致性
- 能够在安全与信任边界上给出可判定结果
