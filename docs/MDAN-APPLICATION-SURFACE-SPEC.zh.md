# MDAN 应用表面规范

- 状态：Draft
- 版本：vNext
- 语言：中文

## 1. 引言

本规范定义 MDAN 应用表面的规范性语义。

本规范的目标是定义一套与具体实现解耦的应用表面模型，使不同实现能够围绕相同的内容语义、动作语义、状态语义和交付语义建立互操作性。

本规范定义：

- JSON 表面封装的规范对象与结构约束
- 内容文档的规范对象与结构约束
- 动作契约的规范对象与结构约束
- 状态绑定表面的语义
- 动作执行、状态迁移与结果分类
- 表示协商、交付边界与一致性要求
- 信任、安全、确认与合规要求

本规范不定义：

- 任意特定 SDK 的 API
- 任意特定编程语言或框架
- 任意特定宿主或运行时的内部实现方式
- 任意特定 UI 组件库或交互外观
- 教程式使用方法、接入步骤或最佳实践

本规范定义的是抽象规范对象及其语义约束，并规定这些对象在不同表示之间必须保持一致的语义边界。

在当前 SDK profile 中，Markdown artifact 是推荐的公开读取表示，HTML 是其浏览器投影；JSON 表面封装仍可作为兼容性交互传输形态存在。不同实现可以选择不同主表示，但不得改变本规范定义的对象边界与语义关系。

## 2. 规范性说明

### 2.1 规范性关键字

本规范中的下列关键字按其规范性含义解释：

- `MUST` / `必须`：满足合规所必需
- `MUST NOT` / `不得`：满足合规所必需的禁止项
- `SHOULD` / `应当`：强烈建议；若偏离，必须有明确理由
- `SHOULD NOT` / `不应`：强烈不建议；若偏离，必须有明确理由
- `MAY` / `可以`：可选能力

### 2.2 解释优先级

若两个规范语句发生冲突，解释优先级如下：

1. 更具体的规则优先于更一般的规则
2. 更严格的安全约束优先于更宽松的安全约束
3. 动作契约中的执行真相优先于内容文档中的叙述性表达
4. 本规范优先于任何实现文档、SDK 文档或示例文档

## 3. 核心术语

### 3.1 JSON 表面封装

`JSON surface envelope` 是一种兼容性交互传输对象。它在一个 JSON 对象中同时承载 `content`、`actions` 以及可选的 `view`。

### 3.2 内容文档

`content` 是面向人类与代理共同阅读的规范表面文本。它定义可读内容、结构化区域以及动作引用语义，但不单独构成完整的执行真相。

### 3.3 动作契约

`actions` 是机器可判定的执行契约。它定义状态身份、动作集合、输入模式、状态影响、允许的后继动作以及默认安全策略。

### 3.4 表面

`surface` 是绑定到某一具体状态的运行时表面。它至少由内容文档与动作语义共同确定，并可以被投影为不同表示形式。

### 3.5 状态

状态是应用某一时刻的身份边界。状态至少由 `app_id`、`state_id` 和 `state_version` 共同标识。

### 3.6 区域

区域是表面中可被单独识别和更新的命名部分。区域更新与整页更新在语义上必须区分。

### 3.7 动作

动作是可执行的状态相关操作。每个动作必须具有稳定身份，并具有明确的目标、输入约束、状态影响和安全语义。

### 3.8 表示

表示是同一状态表面的可交付形式。不同表示可以面向不同消费方，但不得改变同一状态的规范真相。

### 3.9 信任

信任是对某一内容区域解释权限和安全语义的声明。更保守的信任解释优先。

### 3.10 合规

合规是指某一实现满足本规范中对对象结构、状态一致性、执行语义、安全语义和结果分类的最低要求。

### 3.11 术语对照锚点

为后续跨语言版本保持稳定，本规范中的下列中文术语默认对应以下英文锚点：

- JSON 表面封装：`JSON surface envelope`
- 内容文档：`content document`
- 动作契约：`actions contract`
- 表面：`surface`
- 状态：`state`
- 区域：`region`
- 动作：`action`
- 表示：`representation`
- 信任：`trust`
- 合规：`conformance`
- 结果分类：`outcome classification`

## 4. 总体模型

一个合规的 MDAN 应用表面模型由以下核心对象构成：

- 内容文档
- 动作契约
- 状态绑定表面
- 可选的兼容性封装对象（例如 JSON 表面封装）

这些对象之间满足以下不变量：

- 若存在 JSON 表面封装，则它必须同时承载内容文档与动作契约
- 内容文档必须承担共享阅读表面的职责
- 动作契约必须承担执行真相的职责
- 表面必须绑定到一个明确状态
- 状态不匹配不得被静默合并
- 表示差异不得改变同一状态的规范语义

JSON 表面封装、内容文档、动作契约和表面之间是关联关系，而不是互相替代关系。

### 4.1 当前 SDK Profile 与兼容传输

当前 SDK profile 推荐：

- 使用 Markdown artifact 作为公开读取表示
- 使用 HTML 作为浏览器投影表示
- 在需要兼容交互桥接时使用 `application/json` 表示的 JSON 表面封装

JSON 表面封装必须是对象，并且至少包含：

- `content`
- `actions`

JSON 表面封装可以包含：

- `view`

JSON 表面封装可以包含扩展字段。

其中：

- `content` 必须是字符串，承载内容文档
- `actions` 必须是对象，承载动作契约
- `view` 若存在，必须是对象，承载表面附加元数据

扩展字段不得：

- 覆盖 `content`、`actions` 或 `view` 的规范语义
- 放宽动作契约中的安全约束
- 改变状态身份解释
- 改变执行结果分类

当某一实现使用 JSON 表面封装时，它必须满足：

- 内容文档边界可判定
- 动作契约边界可判定
- 状态身份可判定
- 执行真相可判定
- 安全与一致性约束可判定

### 4.2 表示与投影

本规范允许存在辅助表示或投影形式，例如：

- Markdown 可读表示
- HTML 投影表示
- Event Stream 增量表示

不同表示必须满足：

- 不得改变同一状态的执行真相
- 不得改变同一状态的安全约束
- 不得改变同一状态的动作集合
- 不得静默隐藏状态身份冲突

## 5. 内容文档模型

### 5.1 一般要求

内容文档必须是文本形式的规范内容载体。

内容文档可以包含：

- 普通 Markdown 内容
- 结构化 block 区域

内容文档不得单独承担以下职责：

- 定义完整的动作执行真相
- 取代状态身份字段
- 覆盖动作输入模式

### 5.2 Block 区域

内容文档可以声明命名 block 区域。

block 区域的规范形式为：

```md
::: block{id="main" actions="refresh_main,submit_message" trust="untrusted"}
...
:::
```

block 区域满足以下要求：

- `id` 必须存在，且必须是非空字符串
- 同一内容文档内 block `id` 必须唯一
- `actions` 可以省略；若存在，必须是由逗号分隔的动作标识列表
- `trust` 可以省略；若存在，其值必须是 `trusted`、`untrusted` 或 `unknown` 之一
- block 正文不得为空

block 区域中的每个动作引用都必须满足：

- 引用的动作标识必须存在于同状态的动作集合中
- 同一 block 的 `actions` 列表中不得重复引用同一动作标识

### 5.3 语义槽位

语义槽位是内容文档中的提示词结构能力。

语义槽位与代理区块是两套独立机制：

- 语义槽位负责提示词正文的结构
- 代理区块负责 agent-only 内容与人类可见 UI 的隔离

语义槽位不得依赖代理区块包裹才成立。也就是说：

- 页面内容中的语义槽位可以直接存在于普通内容文档中
- block 对应的局部内容也可以直接声明语义槽位
- 是否存在 `agent:begin` / `agent:end` 不改变语义槽位本身的解释规则

本规范承认的槽位名称为：

- `Purpose`
- `Context`
- `Rules`
- `Result`
- `Views`
- `Handoff`

一旦声明支持语义槽位，则其对象满足以下要求：

- 槽位标题必须使用 H2 级别，即 `## <SlotName>`
- 每个承认的槽位在同一内容文档中至多出现一次
- 槽位出现时其正文不得为空
- 页面级内容中，`Purpose`、`Context`、`Rules`、`Result` 为必需槽位
- block 级局部内容中，`Context`、`Result` 为必需槽位
- `Views`、`Handoff` 为可选槽位

若存在 `untrusted` block，其正文不得影响语义槽位的解释。

### 5.4 代理区块

代理区块是内容文档中的 agent-only 可见性边界。

内容文档可以声明代理区块。

代理区块的规范形式为：

```md
<!-- agent:begin id="assistant_context" -->
...
<!-- agent:end -->
```

一旦声明支持代理区块，则其对象满足以下要求：

- 每个 `agent:begin` 必须有且仅有一个对应的 `agent:end`
- 代理区块必须包含非空 `id`
- 同一内容文档中代理区块 `id` 必须唯一
- 代理区块正文不得为空
- 代理区块不得嵌套另一个代理区块

若存在 `untrusted` block，其正文不得影响代理区块的解释。

代理区块一旦出现，必须满足以下运行时语义：

- 代理区块正文仅供 agent 侧消费
- 人类可见 UI、HTML 投影和默认阅读视图必须忽略代理区块正文
- 代理区块是否存在，不得改变页面或 block 的正常交互布局语义
- 代理区块不得作为语义槽位成立的前置条件

### 5.5 内容文档一致性

内容文档中的动作引用满足以下一致性要求：

- block 区域中的结构化动作引用必须指向已声明动作

若内容文档与动作契约不一致，则内容文档不得被解释为覆盖动作契约。

### 5.6 核心与扩展边界

本章中：

- block 区域属于核心规范对象
- 语义槽位属于提示词结构规范对象
- 代理区块属于 agent-only 隔离规范对象

在当前 SDK/runtime 基线中：

- 页面级 surface 可以强制要求页面语义槽位
- block 级 surface 可以强制要求 block 语义槽位
- 代理区块在出现时必须被解析、校验并在人类可见 UI 中忽略

## 6. 动作契约模型

### 6.1 总体结构

动作契约必须是对象。

在当前 vNext profile 中，动作契约承载于 JSON 表面封装的顶层 `actions` 字段中。

动作契约对象的顶层至少必须包含：

- `actions`

动作契约对象可以包含：

- `app_id`
- `state_id`
- `state_version`
- `blocks`
- `actions`
- `allowed_next_actions`
- `security`

### 6.2 状态身份字段

若动作契约声明了状态身份字段，则必须满足：

- `app_id` 必须是非空字符串
- `state_id` 必须是非空字符串
- `state_version` 必须是数值

同一状态表面中的内容、动作和派生表面不得在状态身份字段上互相矛盾。

### 6.3 区域声明

`blocks` 可以用于声明该状态中可识别的区域集合。

若存在 `blocks`，则：

- 它必须是字符串数组
- 其中每个名称都必须在同一文档中表示稳定区域身份
- 区域名称不得重复

### 6.4 动作集合

`actions` 必须是动作对象数组。

每个动作对象必须满足：

- `id` 必须存在，且必须为非空字符串
- `target` 必须存在，且必须为非空字符串
- `id` 在同一动作集合中必须唯一

动作对象可以包含：

- `label`
- `verb`
- `transport`
- `state_effect`
- `guard`
- `security`
- `input_schema`
- `action_id`
- `action_proof`
- `action_issued_at`
- `submit_format`
- `requires_confirmation`
- `submit_example`

### 6.5 动词语义

若动作声明 `verb`，其值必须是以下之一：

- `navigate`
- `read`
- `write`

本规范不为其它动词值声明规范语义。

### 6.6 传输语义

若动作声明 `transport`，则：

- `transport` 必须为对象
- 若存在 `transport.method`，其值必须是 `GET` 或 `POST`

本规范只定义 `GET` 与 `POST` 两类方法的规范语义。

### 6.7 输入模式

若动作声明 `input_schema`，则：

- `input_schema` 必须为对象
- 若存在 `required`，其值必须是字符串数组
- 若存在 `properties`，其值必须是对象

输入模式中的字段语义受以下约束：

- 字段身份由属性名表示
- 必填性由 `required` 表示
- 字段类型可以表达字符串、数字、整数、布尔、枚举、资产、对象、数组等语义
- 格式语义可以进一步细化字段显示或传输约束，但不得与基础类型语义冲突

### 6.8 状态影响

若动作声明 `state_effect`，则：

- `state_effect` 必须为对象
- 若存在 `response_mode`，其值必须是 `page` 或 `region`
- 若存在 `updated_regions`，其值必须是字符串数组

状态影响满足以下约束：

- `page` 表示动作结果语义上更新整页状态
- `region` 表示动作结果语义上只更新被声明的区域集合
- 若声明 `updated_regions`，其中每个区域名称都应当可被同状态表面识别

### 6.9 风险与确认

若动作声明 `guard`，则：

- `guard` 必须为对象
- `guard.risk_level` 若存在，必须表达稳定的风险等级语义

若动作声明 `security`，则：

- `security` 必须为对象
- `security.confirmation_policy` 若存在，其值必须是 `never`、`always` 或 `high-and-above` 之一

### 6.10 默认安全策略

动作契约可以声明顶层默认安全策略：

```json
{
  "security": {
    "default_confirmation_policy": "high-and-above"
  }
}
```

若声明该字段，则：

- `security` 必须为对象
- `default_confirmation_policy` 的值必须是 `never`、`always` 或 `high-and-above` 之一

动作级安全策略优先于文档级默认安全策略。

### 6.11 动作执行凭证与提交格式

动作对象可以携带执行凭证和提交格式提示。

若动作声明 `action_id`，则：

- 其值必须是非空字符串
- 其语义必须稳定指向当前动作执行身份

若动作声明 `action_proof`，则：

- 其值必须是非空字符串
- 其语义必须被限制在当前动作、当前目标、当前方法以及声明的输入集合内
- 它不得被解释为跨状态或跨动作通用凭证

若动作声明 `action_issued_at`，则：

- 其值必须是数值
- 它必须表示该动作凭证或动作声明的签发时间

若动作声明 `submit_format`，则：

- 其值必须是非空字符串
- 它必须表示执行请求期望的输入封装格式

若动作声明 `requires_confirmation`，则：

- 其值必须是布尔值
- 当其值为 `true` 时，执行请求必须携带显式确认事实

若动作声明 `submit_example`，则：

- 其值必须是对象
- 它只能作为提交形状说明，不得覆盖 `input_schema` 的约束

### 6.12 允许的后继动作

`allowed_next_actions` 可以用于约束在当前状态中允许继续执行的动作集合。

若存在该字段，则：

- 它必须是字符串数组
- 每个动作标识都必须存在于当前动作集合中
- 该数组不得引用未知动作

## 7. 表面模型

### 7.1 表面定义

表面是内容文档、动作契约和状态身份共同确定的状态绑定结果。

一个合规表面至少必须满足：

- 具有明确状态身份
- 具有可读内容语义
- 具有可执行动作语义
- 具有与状态影响一致的更新边界

### 7.2 表面种类

本规范承认以下表面结果种类：

- `page`
- `region`
- `stream`

它们必须在语义上相互区分：

- `page` 表示整页状态结果
- `region` 表示区域级状态结果
- `stream` 表示持续交付的增量结果流

实现不得将三者静默折叠为同一结果类别。

### 7.3 表面附加元数据

在当前 vNext profile 中，JSON 表面封装可以通过顶层 `view` 字段携带表面附加元数据。

`view` 若存在，必须是对象。

`view` 可以包含：

- `route_path`
- `regions`

若存在 `view.route_path`，则：

- 其值必须是字符串
- 其语义是当前表面的路由身份
- 它不得改变状态身份或动作执行真相

若存在 `view.regions`，则：

- 其值必须是对象
- 每个键必须是非空区域名称
- 每个值必须是该区域在当前状态中的内容字符串
- 区域结果映射不得包含非字符串区域内容

当前表面的区域集合可以由以下来源共同确定：

- 动作契约中的 `blocks`
- 内容文档中的 block 区域
- `view.regions` 中的区域名称

若这些来源同时出现，则它们必须被解释为同一状态表面的区域声明来源，而不是彼此独立的命名空间。

若某个区域同时出现在多个来源中，其区域身份必须一致。

附加元数据不得改变动作执行真相，仅可补充描述当前状态表面的识别和投影信息。

### 7.4 区域结果与状态影响

动作声明 `state_effect.response_mode = "region"` 时，若结果表面携带 `view.regions`，则 `state_effect.updated_regions` 应当与实际更新的区域结果保持一致。

若 `updated_regions` 声明的区域不存在于结果表面的区域集合中，则消费者不得将该结果静默解释为成功的区域更新。

若无法可靠判定局部区域更新，消费者可以将结果解释为整页表面更新，但不得伪造未返回的区域结果。

## 8. 状态模型

### 8.1 状态身份

状态身份由以下字段共同界定：

- `app_id`
- `state_id`
- `state_version`

它们共同满足以下要求：

- `app_id` 标识应用身份边界
- `state_id` 标识某一具体状态实例
- `state_version` 标识状态版本单调性边界

### 8.2 状态一致性

若同一表面的多个构件都声明了状态身份字段，则这些字段必须一致。

以下情况构成状态不一致：

- `app_id` 不一致
- `state_id` 不一致
- `state_version` 不一致

状态不一致时，实现必须失败关闭，不得静默推断兼容。

### 8.3 陈旧状态

若执行请求基于陈旧状态发起，则实现必须将其视为显式状态问题。

陈旧状态不得被静默接受为最新状态。

## 9. 执行模型

### 9.1 执行请求

执行请求必须至少表达以下语义：

- 目标动作身份
- 输入值
- 请求所绑定的状态上下文

若动作要求显式确认，则执行请求还必须表达确认事实。

### 9.2 动作选择

执行时被选择的动作必须满足：

- 它存在于当前状态的动作集合中
- 若存在 `allowed_next_actions`，则该动作必须在允许集合内
- 输入值不得超出该动作所允许的输入集合

### 9.3 输入有效性

若动作声明输入模式，则执行请求中的输入必须满足：

- 所有必填输入必须存在
- 未声明的输入不得被解释为合法输入
- 输入值必须满足声明类型与约束

### 9.4 确认策略

若动作或文档安全策略要求确认，则未经确认的请求不得被视为成功执行。

`high-and-above` 的确认策略表示：达到高风险阈值及以上的动作必须要求显式确认。

### 9.5 执行结果分类

动作执行结果必须区分以下结果类别：

- `accepted`
- `rejected`
- `errored`

这三类结果不得被折叠为单一成功或失败语义。

其中：

- `accepted` 表示请求满足约束并被接受为有效状态迁移的一部分
- `rejected` 表示请求被规范性规则拒绝
- `errored` 表示执行过程发生非规范性业务失败或系统失败

结果分类满足以下边界：

- `accepted` 只适用于请求已被理解、已通过约束校验且已被采纳的情况
- `rejected` 只适用于请求已被理解但因状态、策略、权限、确认或输入约束而未被采纳的情况
- `errored` 适用于请求未能形成可判定的规范执行，或执行过程中出现非策略性失败的情况

实现不得将以下情况归类为 `accepted`：

- 动作未知
- 状态身份不一致
- 输入违反动作约束
- 必需确认缺失

实现不得将以下情况归类为 `rejected`：

- 请求封装无法被语义化理解
- 表示或载荷格式本身无效
- 运行时内部故障导致无法完成执行

实现不得将以下情况归类为 `errored`：

- 已被理解且仅因策略规则拒绝的请求

### 9.6 状态迁移

若动作执行被接受，则其结果可以：

- 保持原状态
- 产生新的状态
- 产生新的整页表面
- 产生新的区域结果
- 产生流式结果

状态迁移后的表面必须满足新的状态一致性要求。

### 9.7 基线归类

为减少跨实现歧义，符合本规范的实现应当至少满足以下基线归类：

- 未知动作：`rejected`
- 动作不在允许集合内：`rejected`
- 输入模式校验失败：`rejected`
- 必需确认缺失：`rejected`
- 语义有效但状态身份不匹配：`rejected`
- 请求格式无法被语义化理解：`errored`
- 表示协商失败：`errored` 或显式不可接受结果，但不得伪装为 `accepted`
- 运行时或宿主内部失败：`errored`

## 10. 交付与表示模型

### 10.1 表示种类

本规范承认以下表示类别：

- JSON Surface Envelope
- Markdown
- HTML
- Event Stream

当前 SDK profile 以 `Markdown artifact` 作为主交互读取表示；`JSON Surface Envelope`
仅作为兼容桥接表示保留。

其它表示可以服务于不同消费方，但它们必须共同指向同一状态语义。

除 JSON 表面封装外，本规范不要求其它表示必须使用单一固定封装格式；具体封装格式可以由 profile 单独规定。

### 10.2 表示一致性

表示之间必须满足以下要求：

- 不得因表示不同而改变状态身份
- 不得因表示不同而改变动作集合
- 不得因表示不同而改变安全约束
- 不得因表示不同而改变区域更新语义

Markdown artifact 承担共享阅读表面，嵌入式动作声明承担执行真相；HTML 与
JSON Surface Envelope 都是该状态的投影或兼容承载，不得改变其规范语义。

### 10.3 协商

若存在表示协商，则实现必须根据显式协商结果返回对应表示，或显式拒绝不可接受的表示请求。

实现不得在协商失败时静默返回与请求语义冲突的表示。

当前 SDK profile 中，交互消费者请求主交互读取表示时应使用
`text/markdown`。只有在显式需要兼容桥接时，才应请求 `application/json`。

若某一表示类别存在多个合法 profile，则 profile 的选择也必须是显式的、可判定的，或由更高层规范明确规定。

## 11. 信任与安全模型

### 11.1 信任取值

本规范承认以下信任取值：

- `trusted`
- `untrusted`
- `unknown`

### 11.2 信任优先级

当宽松解释与严格解释发生冲突时，必须采用更严格的安全解释。

当可信内容与不可信内容共同影响解释时，不可信内容不得扩大动作能力、放宽输入约束或降低确认要求。

### 11.3 安全闭合原则

在以下情况下，实现必须失败关闭：

- 状态身份不一致
- 动作身份未知
- 输入超出允许范围
- 必需确认缺失
- 内容文档引用未知动作

## 12. 一致性与冲突规则

### 12.1 内容与动作冲突

若内容文档中的叙述性文本与动作契约中的执行真相冲突，则执行真相以动作契约为准。

### 12.2 局部与全局冲突

若动作级规则与文档级默认规则冲突，则动作级规则优先。

### 12.3 安全策略冲突

若多个安全策略同时存在，则更严格的策略优先。

## 13. 合规要求

一个实现若声称符合本规范，至少必须满足：

- 能够识别内容文档中的 block 语义
- 能够验证动作契约中的动作身份、方法、输入模式、状态影响和安全策略
- 能够在同一状态边界内保持身份一致性
- 能够区分 `page`、`region`、`stream` 三类结果语义
- 能够区分 `accepted`、`rejected`、`errored` 三类执行结果
- 在关键安全与一致性冲突下失败关闭

若实现违反上述任何一项，则不得声称符合本规范。

若实现额外声明支持语义槽位或代理区块，则还必须满足第 5.3 节和第 5.4 节中的对应要求。

## 14. 版本与兼容性

### 14.1 规范版本

本规范的版本独立于任何参考实现版本。

### 14.2 向后兼容

新增字段或新增可选能力时，不得改变既有字段的既定规范语义。

### 14.3 不兼容变更

下列变更视为不兼容变更：

- 改变既有字段的核心语义
- 改变既有结果类别的边界
- 改变既有安全策略值的含义
- 改变状态身份字段的解释方式

发生不兼容变更时，必须通过明确的规范版本升级体现。
